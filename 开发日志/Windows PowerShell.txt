  ### 1. ⚠️ Bug 1：日K线除权日期标记错误 & 历史除权标记缺失

  在 update_stock_kline_day.py 中，除权检测和标记逻辑如下：

        if last_date:
            first_row = data_list[0]
            first_date = first_row[0]
            first_preclose = safe_float(first_row[6])
            prev_close = get_previous_close(conn, code, first_date)
            if prev_close is not None and first_preclose is not None:
                if abs(first_preclose - prev_close) > 0.001:
                    dividend_detected = True
                    dividend_date = first_date

  #### 问题分析：

  1. 除权日期标记错误：
  假如您的脚本周末停跑，周一（7月13号）增量更新时拿到了周一和周二（7月14号，除权发生日）的数据。
  由于 Baostock 在触发除权时会对整个查询区间的所有数据进行前复权，因此周一的  preclose  相比数据库已存的上周五  close
  发生了改变。脚本会正确触发  dividend_detected = True 。
  但在标记时，代码直接将除权日设为  dividend_date = first_date （即周一），导致把周一错误标记为了除权日（ is_dividend = 1
  ），而不是真正发生除权的周二。
  2. 全量导入时历史除权全部丢失：
  如果首次运行该脚本，或者清空表后全量初始化，此时  last_date  为  None ，导致  dividend_detected  永远为  False
  。这会让历史几千条数据中所有的除权除息日全部被标记为  0 ，没有一个被正确标记为  1 。

  #### 💡 解决方案：

  我们不应该只对比第一条数据和数据库。真正的除权日在  data_list  中的特征是：当天的  preclose  不等于前一个交易日的
  close 。
  我们可以在拉取到  data_list  后，遍历每一行 t（从第 2 行开始），对比  row[t]['preclose']  和  row[t-1]['close'] ：

    # 遍历判断实际除权日
    for i in range(1, len(data_list)):
        prev_close = safe_float(data_list[i-1][5]) # 前一日 close
        curr_preclose = safe_float(data_list[i][6]) # 当日 preclose
        if prev_close is not None and curr_preclose is not None:
            if abs(curr_preclose - prev_close) > 0.001:
                # 说明 data_list[i] 这一天是除权日

  这个方法既适用于全量导入，也适用于增量导入，且除权日定位百分之百精准。

  ### 2. ⚠️ Bug 2：周/月K线除权检测范围漏检

  在 update_stock_kline_weekly.py （以及月K线脚本）中，确定检测除权开始的范围时，代码是：

        week_start, week_end = get_week_range(target_date)
        check_start = start_date if start_date > week_start else week_start
        has_div = has_dividend_in_range(conn, code, check_start, target_date)

  #### 问题分析：

  假设上一次周线更新是  2026-07-03 （周五），本次更新是下周一  2026-07-13 ，此时：

  •  start_date  为  2026-07-04 。
  •  week_start  为  2026-07-13 。
  • 比较结果： 2026-07-04 > 2026-07-13  为  False ，所以  check_start  变成了  2026-07-13 。
  •  has_dividend_in_range  只会在  [2026-07-13, 2026-07-13]  范围内检查日K线是否有除权。
  • 假如在上周三（ 2026-07-08 ）发生过一次除权，因为它在  check_start  之前，所以被完全漏掉了！
  • 结果是：周K线判定“没有除权”，仅做增量追加，导致库中  2026-07-03
  之前的周K线没有进行复权调整，而新加入的周K线是复权后的价格，周K线数据在新旧拼接处发生严重的数据断层！

  #### 💡 解决方案：

  检测周/月K线除权的开始日期不应该被  week_start  截断，直接使用  start_date （即  last_date + 1
  ）即可。只要在上一次更新后至今的任何一天发生过除权，周/月K线就需要清空重刷：

        has_div = has_dividend_in_range(conn, code, start_date, target_date)

  ### 3. 📉 性能瓶颈：N+1 循环查询导致的延时（TiDB Cloud 跨域访问特别严重）

  1. 日K线脚本的循环查询：
  在 update_stock_kline_day.py 的主循环里，每遍历一只股票，都会去查一遍数据库获取  MAX(date)  还有  get_previous_close
。对于 5000+
  只股票，这就意味着有 10,000 - 15,000 次 数据库独立网络请求！
  2. 8个季频财务脚本的重复查询：
  在 update_stock_profit_quarterly.py 等 8 个季频财务脚本中， main  函数明明已经在开头批量查询了所有股票的最新日期
profit_status
  。但是在主循环里，调用的  update_stock_data  内部第 261 行又去执行了一次  SELECT MAX(stat_date) FROM ... WHERE code =
  %s 。这导致每个季频财务脚本依然产生了 5,000 次 冗余 SQL 查询。

  #### 影响估算：

  如果 GitHub Actions 和 TiDB Cloud 的网络延迟是 50ms：

  • 每次运行日K线更新，循环查询耗时约：10000 × 50 ms = 500 秒（超 8 分钟）。
  • 8 个财务脚本冗余查询耗时约：8 × 5000 × 50 ms = 2000 秒（超 33 分钟）。
  这会导致 GitHub Actions 执行时间极长，甚至经常超时报错，并白白浪费 TiDB 数据库的连接资源。

  #### 💡 解决方案：

  • 日K线/周K线/月K线：
  在  main  循环开始前，用一条 SQL 批量查出所有股票的最新日期及对应的 close：
    SELECT code, date, close
    FROM stock_kline_day
    WHERE (code, date) IN (
        SELECT code, MAX(date)
        FROM stock_kline_day
        GROUP BY code
    )
    然后在内存里构建成  latest_info = {code: (date, close)}  字典，循环中直接读取，实现零循环查询。
  • 8个季频财务脚本：
  直接将  main  已经获取到的  last_date  作为参数传入  update_stock_data(..., last_date) ，删去函数体内重复的 SQL 查询。

  ### 🔍 报错原因分析

  报错信息为： pymysql.err.InterfaceError: (0, '') ，在执行主控脚本的  mark_script_status （记录进度状态）时触发。

  这是长时任务调度中非常经典的数据库空闲连接超时断开 (Connection Idle Timeout) 现象：

  1. 主控脚本  run_all_updates.py  启动时成功创建了数据库连接  conn 。
  2. 随后它启动了子进程运行  update_index_kline_day.py  等长时同步脚本。
  3. 在子进程运行期间，主进程处于挂起等待状态，主进程持有的  conn  连接在十几分钟内没有任何数据往来。
  4. TiDB Cloud 或者是网络网关/NAT 的安全策略会自动切断并强行重置（RST）已经空闲超时的连接（通常超时设置为 5-10 分钟）。
  5. 当子进程执行完毕，主控进程醒来，试图调用  mark_script_status(conn, ...)
  往已死掉的连接中写入“已成功”或“优雅退出”状态时，直接抛出了连接失效异常，导致主程序崩溃。
  ──────
  ### 🛠️ 解决方案与代码更新

  我已经直接修改了 run_all_updates.py 文件：

  1. 采用即开即闭的临时连接策略：
  在  mark_script_status  和  get_task_summary
  这两个贯穿整个循环调度周期的状态读写函数中，不再复用初始化时的那个连接。函数被调用时，会自动建立一个独立的短期连接
  temp_conn ，在执行完 SQL 并  commit  后立刻释放连接。
  这避免了在长达数小时的任务中持有任何空闲连接，实现对网络超时/防火墙拦截的绝对免疫。
  2. 完美支持 Pending 状态重置：
  重构了  mark_script_status  的内部 SQL 判断分支，增加了对  pending  状态更新的支持（将  started_at  和  completed_at
  置为  NULL ）。这可以让子脚本因超时优雅退出时，将其进度重设回待处理状态，以便下次工作流启动时识别。

  ### ❓ 问题 1：为什么  update_stock_operation_quarterly.py  卡在  ------  处没有任何输出？

  原因分析（并没有卡死，只是标准输出缓存了）：

  1. 对比上一脚本：在  update_stock_profit_quarterly.py  运行时，它打印了全部进度并迅速执行完成。这是因为该脚本检测到
  5201 只股票全部已是最新，直接全部跳过，在不到 1 秒钟内执行完毕。进程退出时，Python
  会自动强制刷新所有的输出缓存，所以您看到了完整的进度。
  2. 当前脚本行为：在  update_stock_operation_quarterly.py  运行时，它发现只有  546  只股票有数据，剩余的  4655
  只股票需要实时调用 Baostock 的 API 去拉取数据。由于单只股票 API 请求加上网络延时需要约 0.2~0.5
  秒，处理剩下的几千只股票需要持续运行二三十分钟。
  3. 缓存阻塞：当 Python 脚本以子进程方式运行（stdout 被重定向到管道）时，Python 默认会启用 4KB/8KB
  块缓冲机制。由于进度函数  print_progress  没有加上  flush=True ，且子进程正在运行中，这些进度行全都被积压在 Python
  内部的内存缓冲区中，没有实时吐给主控程序，因此在 GitHub 界面上看起来像是卡死在了  ------
  处，但其实后台一直在正常拉取。

  我们的修复方案：

  1. 子进程强制无缓冲运行：在主控程序 run_all_updates.py 中，我将子进程启动命令由  [sys.executable, script_path]  修改为
  [sys.executable, "-u", script_path] （注入  -u  选项）。这能够强制 Python 以无缓冲区模式运行所有子脚本。
  2. 实时刷新：在公共进度输出函数 utils.py 的  print_progress  中，我也显式加上了  flush=True 。
  这两项改动可以确保未来所有的更新进度和日志，在生成的那一瞬间立刻实时刷新显示在 GitHub Actions 的控制台上。

  ### ❌ 问题 2：五个宏观脚本报错  TypeError: not all arguments converted during string formatting

  原因分析：
  宏观数据脚本（如存款利率、贷款利率、货币供应量等）使用的是批量缓冲写入，但它们的 SQL 生成函数  build_insert_sql()
  返回的是只有单行占位符的静态 SQL 语句（例如  VALUES (%s, %s, %s, %s, %s, %s, %s, %s) ）。
  在写入时，它们却把 43 条或 47 条数据的参数扁平化为了一个包含几百个元素的列表  flat_args  并传入  cursor.execute(sql,
  flat_args) 。这导致占位符（只有 8 个）和参数个数（几百个）严重不匹配，从而触发了 PyMySQL 的格式化异常。

  另外，如果因为写入失败重建了连接，在  finally  块中调用已断开的旧连接  conn.close()  会抛出  Already closed
  异常，进而把真实的报错信息给掩盖掉。

  我们的修复方案：
  我已将 5 个宏观更新脚本：

  •  update_macro_deposit_rate.py
  •  update_macro_loan_rate.py
  •  update_macro_money_supply_month.py
  •  update_macro_money_supply_year.py

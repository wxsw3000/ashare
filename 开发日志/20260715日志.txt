一、 核心优化与修复工作汇总

1. TiDB Cloud 数据库连接策略重构（解决 Lost connection 读超时）
   - 问题表现：GitHub Actions 执行过程中，频繁在初始化或查询阶段报错：`pymysql.err.OperationalError: (2013, 'Lost connection to MySQL server during query (The read operation timed out)')`。
   - 根本原因：主控脚本 `run_all_updates.py` 原本持有长生命周期的持久连接。在并发开启临时连接去进行状态清理、或在漫长的子脚本执行等待期间，由于默认 `autocommit=False` 的隐式事务机制，导致 TiDB 网关（Proxy）单向重置/挂起空闲连接。
   - 优化方案：彻底取消主控脚本的持久共享连接，重构为“按需开闭连接”的自给自足模式。所有 9 个数据库辅助函数（如 `check_table_empty`、`init_daily_task` 等）在调用时独立创建连接，执行并 Commit 后立即显式 Close。这使主控程序在等待子进程运行的数小时中不持有任何连接，彻底免疫了网关空闲超时和事务路由冲突。

2. 控制台输出重构与单股进度追踪优化
   - 问题表现：原有控制台输出偏向宏观汇总，缺少对每一只股票同步详情的微观感知。
   - 优化方案：
     - 重构了日/周/月K线、指数日K线、以及全部 8 个季频财报/业绩快报/预告脚本的主循环。
     - 为每只股票/指数的同步动作增加了处理详情日志输出，格式如：
       `sh.600000 | 写入 25 条数据 | 2025-12-01 ~ 2026-07-14 | 耗时: 0.145s`
       `sh.600001 | 跳过 (已是最新) | 耗时: 0.001s`
       `sh.600002 | 写入 2450 条数据 | 2020-01-02 ~ 2026-07-14 (检测到除权，全量重刷) | 耗时: 1.250s`
     - 为防止大量的详细日志刷屏，同时将主控读取并计算百分比进度条所依赖的 `PROGRESS` 输出频率降低至每 100 只股票汇报一次，兼顾了微观调试的透明度与宏观控制台的整洁美观。

3. 宏观数据脚本批量写入 Bug 修复（TypeError 报错）
   - 问题表现：5 个宏观数据更新脚本在插入数据时报错：`TypeError: not all arguments converted during string formatting`。
   - 根本原因：`cursor.execute(sql, flat_args)` 在批量插入多行时，不符合 PyMySQL 要求的批量插入入参格式。
   - 优化方案：修改 `flush_db_buffer` 为 PyMySQL 标准的 `executemany(sql, batch_data)` 批量执行结构（batch_data 为元组列表），并对 `conn.close()` 进行了安全的 `try-except` 封装。

4. 子进程输出缓冲延迟解决
   - 优化方案：修改主控脚本启动子进程的命令行，使用 Python 的 `-u`（无缓冲）标志运行子脚本：`[sys.executable, "-u", script_path]`，并为公共打印组件补充 `flush=True`，彻底解决了 GitHub Actions 控制台由于块缓冲导致日志积压延迟显示的现象。

5. 启动状态清理与断点续传恢复
   - 优化方案：在主控脚本启动时增加 `auto_reset_running_task` 步骤，自动将上次中断/超时遗留的 `running` 状态脚本重置为 `pending`，修正了可视化进度的准确性，并能完美实现跨 GitHub Action 调度的无缝断点续传。
